# 웹소켓 로드 밸런싱

웹소켓은 오래 지속되는 상태 저장(stateful) 연결을 설정하므로, 상태 비저장(stateless) HTTP 트래픽에 비해 로드 밸런싱에 있어 특별한 과제와 고려 사항이 따릅니다. 효과적인 로드 밸런싱은 여러 서버 인스턴스에 웹소켓 연결을 분산시켜 확장성, 고가용성 및 최적의 성능을 보장하는 데 매우 중요합니다.

## 1. 고정 세션(Sticky Session, 세션 선호도)의 필요성

종종 상태 비저장인 기존 HTTP 요청과 달리, 웹소켓 연결은 본질적으로 상태를 저장합니다. 특정 서버와 웹소켓 핸드셰이크가 완료되면 해당 서버가 연결의 상태와 컨텍스트를 보유하게 됩니다.

*   **상태 저장 연결:** 클라이언트가 서버 A와 웹소켓 연결을 설정하면, 해당 연결에 대한 모든 후속 메시지는 *반드시* 서버 A로 라우팅되어야 합니다. 세션 선호도가 없는 로드 밸런서가 서버 A의 기존 연결을 위한 메시지를 서버 B로 라우팅하면, 서버 B는 해당 연결을 인식하지 못하여 오류가 발생하거나 메시지가 유실됩니다.
*   **고정 세션 (세션 선호도):** 특정 클라이언트의 모든 요청/메시지가 초기에 연결을 처리한 동일한 백엔드 서버로 일관되게 라우팅되도록 보장하는 로드 밸런서 기능입니다.
    *   **작동 방식:** 로드 밸런서는 다음과 같은 다양한 방법으로 고정성을 구현할 수 있습니다.
        *   **소스 IP 해싱:** 동일한 클라이언트 IP의 요청을 동일한 서버로 라우팅합니다. 여러 클라이언트가 단일 NAT 게이트웨이 뒤에 있는 경우 문제가 될 수 있습니다.
        *   **쿠키:** 로드 밸런서(또는 애플리케이션)가 서버를 식별하는 쿠키를 설정합니다. HTTP 기반 초기 연결에 적합하지만, 핸드셰이크 후 순수 웹소켓 연결에는 덜 직접적입니다.
        *   **연결 기반 해싱:** 일부 고급 로드 밸런서는 연결과 서버 간의 매핑을 유지할 수 있습니다.
*   **웹소켓에서의 중요성:** 고정 세션은 일반적으로 연결 무결성을 유지하기 위해 여러 백엔드 서버가 있는 웹소켓 배포에 **필수적**입니다.

## 2. 로드 밸런싱 알고리즘과 웹소켓 적합성

고정 세션은 연결이 한 서버에 유지되도록 보장하지만, *새로운* 웹소켓 연결의 초기 분산은 로드 밸런싱 알고리즘에 의해 결정됩니다.

*   **가. 라운드 로빈 (Round Robin):**
    *   **작동 방식:** 새로운 연결을 서버에 순차적으로 번갈아 분배합니다.
    *   **웹소켓 적합성:** 구현이 간단하며 서버가 동일하게 프로비저닝되고 연결 지속 시간이 비교적 균일한 경우 잘 작동할 수 있습니다. 그러나 기존 서버 부하나 다양한 연결 강도를 고려하지 않습니다. 서버가 이미 기존 웹소켓 연결로 과부하 상태이더라도 새 연결을 받을 수 있습니다.
    *   **고려 사항:** 고정 세션이 지속성을 처리하고 주 목표가 *새로운* 연결의 균등 분배일 때 가장 잘 사용됩니다.

*   **나. 최소 연결 (Least Connections):**
    *   **작동 방식:** 현재 활성 연결 수가 가장 적은 서버로 새 연결을 보냅니다.
    *   **웹소켓 적합성:** 이것은 종종 **웹소켓에 가장 적합한 알고리즘**입니다. 웹소켓은 오래 지속되므로 "최소 연결"은 서버 전체에 영구적인 부하를 균등하게 분산하여 단일 서버가 너무 많은 동시 연결로 압도되는 것을 방지하는 데 도움이 됩니다.
    *   **고려 사항:** 로드 밸런서가 각 서버의 활성 연결 수를 파악할 수 있어야 합니다.

*   **다. 최소 응답 시간 (Least Response Time):**
    *   **작동 방식:** 현재 가장 빠르게 응답하는 서버로 새 연결을 보냅니다 (응답 시간을 측정하는 상태 확인 필요).
    *   **웹소켓 적합성:** 유용할 수 있지만, 웹소켓의 "응답 시간" 메트릭은 빠른 HTTP 응답보다는 진행 중인 메시지 트래픽을 처리하는 서버의 용량과 더 관련될 수 있습니다. "응답 시간"이 웹소켓 처리 상태를 정확하게 반영한다면 효과적일 수 있습니다.
    *   **고려 사항:** 웹소켓 서버의 "응답 시간" 정의 및 측정을 신중하게 고려해야 합니다.

*   **라. IP 해시 (IP Hash):**
    *   **작동 방식:** 클라이언트 IP 주소의 해시를 기반으로 서버가 선택됩니다. 이는 본질적으로 고정성을 제공합니다.
    *   **웹소켓 적합성:** 로드 밸런서에 명시적인 고정 세션 기능 없이 세션 선호도를 달성하는 간단한 방법이 될 수 있습니다. 그러나 많은 클라이언트가 제한된 IP 주소 집합(예: 동일한 회사 네트워크의 사용자)에서 발생하는 경우 불균등한 부하 분산으로 이어질 수 있습니다.
    *   **고려 사항:** 서버가 다운되면 덜 유연합니다. 해당 서버로 해시된 모든 클라이언트는 연결이 끊어지고 재연결 시 여러 다른 서버로 다시 해시될 수 있어 연쇄적인 문제(thundering herd)를 일으킬 수 있습니다.

*   **마. 가중 알고리즘 (Weighted Round Robin, Weighted Least Connections):**
    *   **작동 방식:** 관리자가 서버 용량에 따라 서버에 다른 가중치를 할당할 수 있습니다(예: 더 강력한 서버가 더 높은 가중치를 받음).
    *   **웹소켓 적합성:** 백엔드 서버의 리소스 용량이 다른 이기종 환경에서 유용합니다. 더 정교한 밸런싱을 위해 "최소 연결"과 결합할 수 있습니다.

**권장 사항:** 대부분의 웹소켓 사용 사례에서는 **최소 연결** (초기 연결 분배용)과 **명시적인 고정 세션 지원** (선택된 서버에서 연결 유지용)을 결합하는 것이 선호되는 접근 방식입니다.

## 3. 웹소켓 서버 확장

증가하는 동시 사용자 수와 메시지 볼륨을 처리하기 위해 웹소켓 애플리케이션은 확장 가능해야 합니다.

### 가. 수평적 확장 (Scaling Out)

*   **개념:** 부하를 분산하기 위해 더 많은 서버 인스턴스를 추가합니다. 이는 웹소켓 애플리케이션 확장을 위한 가장 일반적이고 일반적으로 선호되는 방법입니다.
*   **요구 사항:**
    *   **로드 밸런서:** 여러 인스턴스에 들어오는 연결을 분산하는 데 필수적입니다.
    *   **고정 세션:** 논의된 바와 같이 웹소켓 상태 저장에 매우 중요합니다.
    *   **공유 상태 관리 (필요한 경우):** 애플리케이션 상태를 서로 다른 웹소켓 서버 간에 공유하거나 액세스해야 하는 경우(예: 서로 다른 인스턴스에 연결된 사용자에게 메시지 브로드캐스팅 또는 애플리케이션 수준 사용자 현재 상태), 외부 공유 데이터 저장소 또는 메시징 시스템이 필요합니다.
        *   **예시:**
            *   **Redis Pub/Sub:** 여러 웹소켓 서버 인스턴스 간에 메시지를 브로드캐스팅하는 데 일반적으로 사용됩니다. 서버 인스턴스가 다른 인스턴스에 연결된 클라이언트에게 메시지를 보내야 할 때 해당 메시지를 Redis 채널에 게시합니다. 모든 서버 인스턴스는 이 채널을 구독하고 메시지를 수신한 다음 관련 연결된 클라이언트에게 전달합니다.
            *   **RabbitMQ 또는 Kafka:** 복잡한 서버 간 통신을 위한 더 강력한 메시징 큐입니다.
            *   **공유 데이터베이스:** 영구적인 애플리케이션 상태를 저장합니다.
*   **장점:**
    *   향상된 내결함성 (한 서버가 실패하면 다른 서버가 인계받을 수 있음).
    *   더 많은 시스템을 추가하여 점진적으로 확장하기가 더 쉽습니다.
*   **과제:**
    *   분산 상태 관리의 복잡성.
    *   고정 세션이 올바르게 구성되었는지 확인합니다.

### 나. 수직적 확장 (Scaling Up)

*   **개념:** 단일 서버의 리소스(CPU, RAM, 네트워크 용량)를 늘립니다.
*   **웹소켓 적용 방식:** 단일의 강력한 서버는 특히 애플리케이션과 웹소켓 서버 소프트웨어가 고도로 최적화된 경우 상당한 수의 웹소켓 연결을 처리할 수 있습니다.
*   **장점:**
    *   더 간단한 인프라 (초기에는 복잡한 로드 밸런싱이나 분산 상태 관리가 필요 없음).
*   **한계:**
    *   **단일 장애 지점 (Single Point of Failure):** 서버가 다운되면 전체 애플리케이션을 사용할 수 없게 됩니다.
    *   **물리적 한계:** 단일 서버를 확장할 수 있는 상한선이 있습니다.
    *   **비용:** 고급 서버는 매우 비쌀 수 있습니다.
    *   **업그레이드를 위한 다운타임:** 확장은 종종 다운타임을 필요로 합니다.
*   **적합성:** 소규모 애플리케이션이나 시작점으로 적합할 수 있습니다. 그러나 고가용성 및 대규모 확장을 위해서는 일반적으로 수평적 확장이 필요합니다. 종종 애플리케이션은 수직적 확장으로 시작한 다음 성장함에 따라 수평적 확장으로 이동하거나 결합합니다.

## 4. 로드 밸런서 유형 및 고려 사항

*   **계층 4 (전송 계층) 로드 밸런서:**
    *   TCP/UDP 수준에서 작동합니다.
    *   웹소켓 (TCP) 연결을 분산할 수 있습니다.
    *   고정성은 종종 IP 주소 또는 TCP 세션 정보를 기반으로 합니다.
    *   패킷 내용을 검사하지 않으므로 일반적으로 더 빠릅니다.
*   **계층 7 (애플리케이션 계층) 로드 밸런서:**
    *   HTTP 헤더를 검사할 수 있어 초기 HTTP 기반 웹소켓 핸드셰이크 및 잠재적으로 쿠키 기반 고정성에 더 유연합니다.
    *   SSL/TLS (WSS)를 종료하여 백엔드 서버에서 암호화 작업을 오프로드할 수 있습니다.
    *   일부 최신 계층 7 로드 밸런서는 웹소켓 프록시에 대한 명시적인 지원을 제공하며 고정성을 더 지능적으로 유지할 수 있습니다.
*   **소프트웨어 대 하드웨어 로드 밸런서:**
    *   **소프트웨어:** Nginx, HAProxy, Envoy. 유연하고 비용 효율적이며 클라우드 환경에서 일반적으로 사용됩니다.
    *   **하드웨어:** 전용 어플라이언스 (예: F5 BIG-IP, Citrix ADC). 매우 높은 성능을 제공할 수 있지만 일반적으로 더 비쌉니다.
*   **상태 확인 (Health Checks):** 로드 밸런서는 백엔드 웹소켓 서버에 대한 상태 확인을 수행하여 정상 인스턴스로만 연결을 라우팅하도록 해야 합니다. 상태 확인에는 간단한 TCP 연결 테스트 또는 서버의 웹소켓 트래픽 처리 능력을 반영하는 특정 HTTP 엔드포인트가 포함될 수 있습니다.

## 결론

웹소켓을 효과적으로 로드 밸런싱하려면 상태 저장 연결을 유지하기 위해 세션 선호도(고정 세션)에 세심한 주의를 기울여야 합니다. "최소 연결" 알고리즘은 종종 새 연결을 분산하는 데 선호됩니다. 강력한 로드 밸런싱 설정과 Redis Pub/Sub과 같은 공유 메시징 시스템의 지원을 받는 수평적 확장은 웹소켓을 사용하여 확장 가능하고 탄력적인 실시간 애플리케이션을 구축하는 표준 접근 방식입니다.
